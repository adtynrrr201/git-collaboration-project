## üöÄ Getting Started

1. Fork the repository
2. Clone your fork locally
3. Create a new branch for your feature or bug fix
4. Make your changes
5. Test your changes thoroughly
6. Commit your changes with a descriptive commit message
7. Push your changes to your fork
8. Submit a Pull Request

### Setup Instructions
```bash

git clone https://github.com/your-username/git-collaboration-project.git
cd git-collaboration-project

# Create a new branch for your contribution
git checkout -b feature/your-feature-name
```

## ü§ù How Can I Contribute?

### Reporting Bugs
Before creating a bug report, please check the issue list to see if the problem has already been reported. When creating a bug report, include:
- A clear title and description
- Steps to reproduce the issue
- Expected vs. actual behavior
- Environment information (OS, browser, etc.)

### Suggesting Enhancements
Enhancement suggestions are welcome! When suggesting enhancements, please:
- Provide a clear and detailed explanation of the feature
- Describe the current behavior and explain the suggested improvement
- Explain why this enhancement would be useful

### Code Contributions
- Look through the existing issues for items marked "good first issue" if you're new to the project
- Fork the repository and create your branch from `main`
- Follow the coding standards used throughout the project
- Add tests if your changes require them
- Ensure all tests pass before submitting your pull request

## üåø Branch Naming Convention

Use the following convention for branch names:
- `feature/short-description` - for new features
- `bugfix/short-description` - for bug fixes
- `hotfix/short-description` - for urgent fixes
- `docs/short-description` - for documentation updates
- `refactor/short-description` - for refactoring changes
- `test/short-description` - for adding or updating tests
- `chore/short-description` - for maintenance tasks

Example: `feature/add-contribution-guidelines`, `bugfix/typo-in-readme`

## ‚úçÔ∏è Commit Message Guidelines

Follow these guidelines for writing commit messages:

- Use the present tense ("Add feature" not "Added feature")
- Use the imperative mood ("Move cursor to..." not "Moves cursor to...")
- Limit the first line to 72 characters or less
- Reference issues and pull requests after the first line
- Capitalize the first letter of the subject line
- Separate the subject from the body with a blank line (for longer commits)

Examples:
```
feat: Add contribution guidelines document

This adds a comprehensive guide for contributors to follow
when making changes to the repository. It covers branch naming,
commit message format, and the pull request process.
```

```
docs: Update README with installation instructions

- Add prerequisites section
- Include step-by-step installation guide
- Add troubleshooting tips
```

## üîÑ Pull Request Process

1. Fill in the Pull Request template (if available)
2. Follow the naming convention: `[Type] Brief description of changes`
3. Link any relevant issues
4. Include screenshots if UI changes are involved
5. Ensure all tests pass
6. Update documentation as needed
7. Request reviews from maintainers

### Pull Request Title Format
- `feat:` Add new feature
- `fix:` Correct issue
- `docs:` Update documentation
- `style:` Formatting changes
- `refactor:` Code restructuring
- `test:` Add or update tests
- `chore:` Maintenance tasks

### Before Submitting
- Ensure your code follows the project's style guidelines
- Run all tests to make sure they pass
- Update documentation as needed
- Squash commits if necessary to create a clean history

## üëÄ Code Review Process

- At least one maintainer must approve the PR
- Address all review comments before merging
- Keep PRs focused on a single topic or related changes
- Large changes should be broken into smaller, manageable PRs
- Be respectful and constructive in reviews
- Reviewers should provide clear feedback and suggestions

## üìù Style Guides

### Markdown Style Guide
- Use consistent heading levels
- Use proper code fencing with language identifiers
- Use relative links when referencing other documents in the repository
- Maintain consistent formatting throughout

### Git Style Guide
- Write clear, descriptive commit messages
- Keep commits focused on a single change
- Use the present tense in commit messages
- Sign off on commits when required
